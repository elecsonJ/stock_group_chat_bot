# 멀티-에이전트 동적 토론 프로세스 (Dynamic Multi-Agent Debate Workflow)

이 문서는 AI 주식 투자 봇이 복수의 프론티어 수학/추론 모델과 로컬 판사 모델(gpt-oss-20b)을 활용해 **어떻게 다단계 심층 토론을 진행하면서 최종 결론에 도달하는지(8단계 프로세스)**를 상세히 설명하는 아키텍처 문서입니다.

## 핵심 개념 (Core Concepts)
1. **블라인드 1라운드**: 초기 편향을 막기 위해 타 모델의 의견을 모르는 상태에서 각자 독립적인 분석을 수행.
2. **실시간 스피드 난타전**: asyncio 동시성을 활용해 특정 주장에 대해 **가장 빨리 맹점을 찌르고 반론을 도출한 모델**에게 우선권을 주는 다이나믹 토론 구조.
3. **CoT 공개 및 팩트 체크**: 내부 사색과 논리 전개 과정을 숨기지 않고 `<thought>` 태그 내에 기록하여, 타 모델과 로컬 검증기가 이를 분석할 수 있게 함.
4. **로컬 판사의 다수결 및 교착 판정**: 토론의 결론이 겉기에 나지 않을 경우, 프론티어 API 토큰이나 시간을 낭비하지 않고 자체 로컬 모델이 의견 일치/불합치/다수결/1:1:1 교착 상태를 JSON으로 정리.

## 🔄 8단계 상세 토론 프로세스

### Step 0: RAG 메모리 연동 (과거 결론 회상)
- `rag_agent.py`가 가동되어 사용자의 주제에서 키워드를 추출한 뒤, 과거에 저장된 판결문 및 요약본에서 일치하는 기록을 찾습니다.
- 이 기록은 `[RAG 과거 토론 맥락 사전지식]`이라는 이름으로 토론장의 가장 기초 베이스 라인(base logic)으로 주입됩니다.
- 방어 메커니즘 (Anti-anchoring): 과거 결론에 도출된 내용이 최신 팩트와 맞지 않을 경우, 이어지는 자동 리서치 국면(Phase 1)에서 가져온 최신 데이터를 더 우선시하여 과거 편향을 차단합니다.

### Step 1: 자동 리서치 통계 추출 및 블라인드 초기 주장 수집 (Blind Opening)
- 주제 성격을 분석해 '최신 이슈'일 경우 방대한 프리미엄 뉴스를 스크래핑해 오고, 주식 티커/검색어를 분석하여 수치 중심의 `Fact-Sheet`를 구성합니다.
- `gpt-5.2-2025-12-11`, `claude-sonnet-4-6`, `gemini-3.1-pro-preview` 세 모델이 이 통계 자료와 사전지식을 전달받습니다.
- 서로의 의견이나 존재를 모르는 상태(`asyncio.gather` 병행 호출)에서 각각 **<thought>** 기반의 독립적인 3~4문장 초기 주장을 도출합니다.
- **[핵심 보안 처리]**: 이때 AI가 내뱉은 속마음(`<thought>`) 토큰은 디스코드 사용자에게 시각적으로만 보여주고, 봇들끼리 대화하는 공통 `history` 안에서는 정규식으로 깨끗하게 도려내어 컨닝 방지 및 토큰 낭비를 막습니다.
- 추출된 초기 주장은 모두 모여 다음 라운드(Step 2)를 위한 기록(History)으로 병합됩니다.

### Step 2: 실시간 난타전 - 속도 기반 동적 반론 (Dynamic Speed Round)
- 모든 기록(A, B, C의 의견)을 통합하여 세 모델에게 동시에 전달하고 반론을 요청합니다.
- **[D 도출]**: 세 모델 중 **가장 먼저 반론을 도출(FIRST_COMPLETED)**해 낸 모델 1명의 응답을 채택하고 나머지 태스크는 취소합니다. 이 반론(D)을 기록에 추가합니다.
- **[E 도출]**: 나머지 두 모델을 대상으로 업데이트된 기록(A,B,C,D)을 주며 다시 경쟁시킵니다. 가장 먼저 도출한 1명의 의견(E)을 기록에 추가합니다.
- **[F 도출]**: 남은 1명의 모델이 전체 의견(A,B,C,D,E)을 보고 최종 방어 및 정리 반론(F)을 숙고하여 작성합니다.
- (토론이 교착 상태에 빠지면 이 Step 2 난타전 사이클이 최대 2번까지 반복될 수 있습니다.)

### Step 3: 최종 의견 조율 및 결론 도출 (Consensus Building)
- 1, 2라운드를 거친 전체 히스토리(A~F)를 바탕으로 세 모델 모두가 각자의 1문장 최종 결론을 내놓습니다.
- 글의 맨 마지막에 반드시 `[결론: 찬성]`, `[결론: 반대]`, 또는 `[결론: 중립]` 키워드를 강제하여 확실한 스탠스를 표시하도록 합니다.

### Step 4 & Step 6: 1차/2차 만장일치 확인 (Local Judge Verdict)
- 세 모델의 최종 결론(Step 3)이 도출되면 **로컬 수석 투자 판사 (gpt-oss-20b)**가 투입됩니다.
- 로컬 판사가 텍스트를 분석하여 3명의 스탠스가 모두 일치하는지(`만장일치`) 아니면 엇갈리는지(`불합치`) 판단하여 JSON 포맷으로 평가를 반환합니다.

### Step 5 & Step 7: 의견 불합치에 따른 재개 (Retry Loop)
- **만장일치 성공 시 (Step 5-1, 7-1)**: 로컬 판사의 JSON 판결문을 출력하고, DB에 결과를 저장한 뒤 즉시 토론을 종료합니다.
- **의견 불합치 발생 시 (Step 5-2)**: 토론이 타협점을 찾지 못했다고 판단하여, 다시 **Step 2(난타전)와 Step 3(결론 도출)**를 한 사이클 더 반복합니다. 

### Step 8: 최후 대립 양상 판결 (Deadlock Resolution)
- 최대 2번의 난타전 사이클(총 4~6번 이상의 반박 교환)에도 만장일치에 도달하지 못하면 무한 루프를 방지하기 위해 로컬 판사가 최후 개입합니다.
- 수석 판사(로컬 모델)는 각 모델의 다수결 양상을 파악하고 두 가지로 최종 판단을 내립니다:
    1. **2:1 결론**: 2명이 동의하는 과반수의 핵심 요지와, 1명의 소수 반대 요지를 요약하여 리포트합니다.
    2. **1:1:1 보류 (교착)**: 3명이 완벽히 각자 다른 스탠스(찬성1, 반대1, 중립1 등)를 유지할 경우, 어떤 점에서 대립하는지 3줄로 요약하고 "관망(시장 관전)"을 권고합니다.
- 작성된 모든 토론 로그와 JSON 형식의 최종 판결 리포트는 영구적인 RAG(Retrieval-Augmented Generation) 지식베이스 활용을 위해 SQLite 데이터베이스에 기록됩니다.

## 📈 장점 및 도입 효과
## 📈 장점 및 구조적 탄탄함(Roboustness) 도입 효과
- **빠른 판단력**: 단순 API 순차 호출(A->B->C->A)에 비해, 빠르고 날카로운 논제를 던지는 순서대로 의견이 반영되므로 사람의 딥 토론과 유사한 현실적인 긴장감이 부여됩니다.
- **토큰 누수 원천 차단 & 자율성**: 매 턴마다 세 모델에게 일일이 고정된 발언권을 주는 대신, 논리가 즉각 튀어나오는 모델의 말을 우선 처리하고, `<thought>` 태그를 도려내는 토큰 최적화 과정을 거쳐 API 폭발(Context Limitation)을 방지했습니다.
- **DB Lock / Timeout 프리징 극복**: 동시 접속 시 발생할 수 있는 SQLite Lock 문제(`timeout=20`)를 처리하고, 서버 과부하를 대비한 `asyncio` 기반의 3회 재도전(Retry Loop) 로직을 탑재해 강인한 구동 안정성을 자랑합니다.

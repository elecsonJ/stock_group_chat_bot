# 멀티 에이전트 토론 프로세스 (Current Runtime Spec)

이 문서는 **현재 코드 기준 실제 동작**을 설명합니다.  
실험 아이디어(미구현)는 `workflow_advanced_quant_agent.md`에 분리되어 있습니다.

## 핵심 원칙
1. 블라인드 1라운드로 초기 편향을 줄입니다.
2. 2라운드부터는 고정 순서(GPT→Claude→Gemini) 교차 반박을 수행합니다.
3. `[SEARCH: ...]` 요청 시 Evidence 패키지(URL/발췌/도메인/제약)가 생성되고 DB에 저장됩니다.
4. 내부 사고(CoT) 노출 없이 공개 발언만 토론 히스토리에 반영됩니다.
5. `[ACK]`가 전 모델에서 감지되면 루프를 조기 종료하고 판사 최종 판결로 전환합니다.
6. `[조준:Model]`이 감지되면 타깃 모델에게 즉각 방어 발언권을 부여합니다.
7. 로컬 모델 장애 시 강등 모드로 전환해 규칙 기반 집계를 유지합니다.

## 단계별 흐름
### Step 0: 온톨로지 + RAG 준비
1. 온톨로지 플래너가 질의를 분석해 `mode`, `tickers`, `web_queries`, `rag_keywords`를 생성합니다.
2. 로컬 모델이 정상인 경우 RAG를 통해 과거 토론 맥락을 주입합니다.
3. 로컬 모델 장애 시 RAG는 건너뛰고 휴리스틱 추출기로 대체합니다.
4. `!포트폴리오` 또는 `data/my_portfolio.md`가 있으면 사용자 포트폴리오 컨텍스트를 토론에 주입합니다.

### Step 1: Fact-Sheet 구성
1. 온톨로지 플랜 결과 + LLM 추출 결과를 병합해 ticker/search 목록을 구성합니다.
2. 마스터 파이프라인이 거시/주가/뉴스 요약 데이터를 수집해 공통 Fact-Sheet를 만듭니다.
3. 이상치(배당률, PER, 기관보유율 등)는 경고 플래그로 표시됩니다.

### Step 2: 토론 라운드
1. Loop 1은 블라인드 병렬 발언입니다.
2. `DEBATE_SPEED_MODE=first_completed`면 Loop 1은 FIRST_COMPLETED 1개 발언만 채택합니다.
3. Loop 2+는 순차 교차 반박입니다.
4. 발언에 `[SEARCH: ...]`가 있으면 심층 리서치가 실행됩니다.
5. 리서치 근거에는 세션 전역 `EVxxxx` 근거ID가 부여됩니다.
6. 동일 검색어는 세션 단위 dedup 처리됩니다.
7. 새 증거가 없는 반복 루프는 조기 종료될 수 있습니다.

### Step 3: 최종 결론
1. 3개 모델이 동시 최종 결론을 냅니다.
2. `[최종 선택: ...]` 태그를 포함해야 합니다.
3. `[근거ID: EVxxxx, ...]` 태그를 포함해야 하며 누락 시 자동 복구 재요청합니다.
4. 만장일치 판정 전에 SEARCH 근거가 0회면 시스템이 강제 리서치를 1회 수행합니다.

### Step 4: 판정
1. 1차 판정은 규칙 기반(라벨 정규화) 우선입니다.
2. 규칙 판정 불가 시 로컬 모델 JSON 판정으로 보완합니다.
3. JSON 파싱 실패 시 복구 프롬프트 재시도 후 최종 판정을 내립니다.
4. 로컬 모델 장애 시 규칙 기반 다수 집계로 강등 판정을 수행합니다.

## 출력/저장
1. 토론 로그는 `debates.full_log`에 저장됩니다.
2. Evidence 패키지는 `research_evidences`에 저장됩니다.
3. 요약은 `summaries`에 일/주/월 단위로 저장됩니다.
